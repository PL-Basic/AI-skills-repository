# Learn By Building

`learn-by-building` 是一个面向编程学习和项目实践的 skill。它关注的不是让 AI 更快给出完整代码，而是帮助学习者在真实需求中练习“如何从问题走到实现”。

很多人在学习编程时，会遇到一种很常见的断层：能看懂别人写好的代码，也能跟着教程完成步骤，但一旦面对一个新需求，就不知道应该先分析什么、要改哪些文件、每一层应该承担什么职责。这个 skill 的目标就是补上这段能力。

## 它解决什么问题

`learn-by-building` 主要解决的是学习过程中的“实现判断”问题。

它不会把学习者直接带到最终答案，而是把重点放在需求理解、结构拆分和实现路线判断上。比如一个功能到底是新增、修改、查询、删除，还是一个状态切换；它应该涉及 Controller、Service、DTO、Entity、Mapper，还是还需要前端页面和接口封装；哪些字段应该由用户输入，哪些字段应该由系统生成。

这些判断能力，往往比某一段具体代码更重要。因为真正写项目时，代码不是孤立存在的，它总是落在已有结构、已有约定和已有业务流程里。

## 它的工作方式

这个 skill 会倾向于让学习者先参与思考，再看到参考实现。

它通常会先引导学习者复述需求，明确这个功能解决什么问题、谁会使用、输入和输出是什么。接着再一起拆实现路线，判断要改哪些层、先做什么、后做什么。等路线清楚之后，再把业务步骤写成伪代码，最后才进入参考代码、对比和审查。

这种方式会比直接生成代码慢一点，但更适合学习。因为它训练的是迁移能力：下一次遇到类似需求时，学习者不只是记得一段代码，而是知道应该怎么开始分析。

## 适合什么场景

`learn-by-building` 适合用在边做项目边学习的场景中，尤其适合这些情况：

- 想学习如何从需求拆出实现步骤；
- 想理解一个功能为什么要分 Controller、Service、DTO、Mapper 等层；
- 想在写代码前先确认自己的实现路线是否合理；
- 想把大段代码拆成能理解的业务步骤；
- 想通过代码审查发现自己漏掉的校验、异常处理和边界条件；
- 想减少对“直接复制 AI 答案”的依赖。

它不追求把每个问题都变成教学课件，而是尽量贴着真实项目推进：先理解项目，再分析需求，再写代码。

## 核心价值

这个 skill 的核心价值，是把 AI 从“代码答案提供者”调整为“学习过程陪练者”。

它更关心学习者有没有形成自己的判断：这个需求属于哪类功能，应该影响哪些层，业务顺序是否合理，代码放在这里是否符合项目结构，参考实现和自己的思路有什么差别。

如果说普通的代码生成更像是直接给出结果，那么 `learn-by-building` 更像是在结果之前多保留一段思考过程。它希望学习者最终获得的不是某一次功能的答案，而是下一次独立实现功能的能力。
